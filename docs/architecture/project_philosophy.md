# Project Philosophy

## Назначение

Ophir — это не приложение для учета расходов.

Ophir — это персональный финансовый ассистент. Его главная цель — помогать
пользователю принимать лучшие финансовые решения, а не просто хранить список
операций.

Финансовые данные в Ophir должны превращаться в понятную картину: что
происходит с деньгами, почему это происходит, что может произойти дальше и
какое действие пользователю рекомендуется сделать.

## Главный принцип

Любая часть продукта должна оцениваться не по тому, сколько данных она хранит
или показывает, а по тому, насколько она помогает пользователю принимать
решения.

Операции, категории, бюджеты и цели — это источники данных для финансового
ассистента, а не конечная цель приложения.

## Dashboard

Dashboard — это главный экран финансового ассистента, а не список виджетов.

Он должен помогать пользователю быстро понять финансовое состояние и следующий
разумный шаг. Каждый блок Dashboard должен иметь ясную роль в ответе на один
из пользовательских вопросов:

- Что происходит с моими деньгами?
- Почему это происходит?
- Что произойдет дальше?
- Что мне рекомендуется сделать?

## Вопросы, на которые отвечает feature

Каждая feature в Ophir должна отвечать хотя бы на один из ключевых вопросов
пользователя:

- Что происходит с моими деньгами?
- Почему это происходит?
- Что произойдет дальше?
- Что мне рекомендуется сделать?

Если feature не помогает ответить ни на один из этих вопросов, она не должна
считаться продуктово обоснованной.

## Будущие функции

Все будущие функции должны строиться вокруг философии персонального
финансового ассистента:

- Budget;
- Goals;
- Statistics;
- Cash Flow;
- AI;
- Forecast.

Эти направления не должны развиваться как изолированные экраны учета. Они
должны помогать пользователю понимать ситуацию, видеть причины, оценивать
будущее и выбирать действие.

## Проектирование feature

При проектировании новой feature сначала определяется, какую пользу она
приносит пользователю, и только потом проектируется UI.

Правильный порядок работы:

1. Определить пользовательскую пользу.
2. Определить, на какой ключевой финансовый вопрос отвечает feature.
3. Определить необходимые данные и доменную логику.
4. Спроектировать UI как способ ясно донести решение или рекомендацию.

UI не должен становиться самостоятельной причиной для feature. Интерфейс
служит финансовому смыслу, а не заменяет его.

## Инженерные следствия

Простота интерфейса не означает примитивную модель данных. Domain layer должен
поддерживать расширяемые вычисления: running balance, группировки, периоды,
прогнозы, бюджеты, цели, финансовое здоровье и рекомендации.

Быстрое локальное решение не принимается, если оно ухудшает расширяемость
системы. Любая feature должна быть пригодна для повторного использования,
тестирования и дальнейшего развития.

Повторяющиеся значения, формулы, правила UI, mapping и tokens должны жить в
централизованных слоях. Дублирование допустимо только как временное решение,
зафиксированное в документации и техническом долге.

## Запрещено

- Считать Ophir приложением для учета расходов.
- Добавлять feature только потому, что ее легко реализовать.
- Проектировать UI до определения пользовательской пользы.
- Делать Dashboard набором независимых виджетов без общей ассистентской роли.
- Считать операции, категории, бюджеты или цели конечной целью продукта.
- Принимать архитектурные решения без фиксации в `docs/`.
