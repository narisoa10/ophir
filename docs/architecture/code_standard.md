# Code Standard

## Назначение

Code Standard фиксирует правила написания кода Ophir. Цель стандарта -
сохранять архитектурные границы, переиспользуемость и предсказуемость проекта.

Этот документ задает общие правила кода. Детальные правила операций находятся в
`operation_architecture.md`, domain ownership - в `domain_model.md`, а visual tokens - в Theme, Typography, Spacing и
Color standards. Если правило относится к конкретной подсистеме, владельцем
является специализированный документ.

## Основные правила

### UI не содержит бизнес-логику

UI отвечает за отображение и события. Он не считает balances, totals, category
analytics и не принимает финансовые решения.

### Формулы находятся в Domain

Любая формула, влияющая на деньги, статистику, budget, forecast или financial
health, должна находиться в Domain.

### Presentation готовит данные для UI

Presentation превращает domain data в display-ready models: labels, formatted
amounts, grouped rows, visual states.

### Repository работает только с данными

Repository загружает, сохраняет и мапит данные. Он не знает про visual hierarchy
и не форматирует UI.

### Один файл = один class

Файл должен содержать одну основную сущность. Исключения допустимы только для
маленьких private value objects, тесно связанных с основной сущностью.

### Использовать const

`const` используется везде, где значение может быть compile-time constant. Это
снижает лишние rebuilds и делает намерение явным.

## Theme и Typography

Запрещено:

- создавать feature-specific Theme;
- создавать feature-specific Typography;
- использовать raw color, spacing, radius, font size или font weight;
- копировать tokens в локальные helpers.

Все визуальные значения идут через Theme.

## Переиспользуемость

Любое решение должно быть пригодно для повторного использования. Если поведение
кажется специфичным для feature, нужно проверить, не является ли оно общей
ролью для будущих Dashboard, Statistics, Budget или Settings.

## Как использовать

Перед созданием файла нужно определить:

- слой: UI, Presentation, Domain или Repository;
- ответственность class;
- какие зависимости допустимы;
- есть ли уже похожая сущность;
- нужно ли обновить документацию.

## Как расширять

Новые standards добавляются в этот документ до массового изменения кода. Если
правило меняет архитектурный подход, создается ADR.

## Запрещено

- Смешивать слои ради скорости.
- Дублировать financial calculations.
- Передавать database models напрямую в widgets.
- Добавлять локальные exceptions без документации.
- Исправлять архитектурную проблему styling-хаками в UI.

## Текущее ограничение

Существующий код может содержать legacy patterns. Они не являются прецедентом
для новых features. При изменении legacy участка разработчик должен либо
привести его к стандарту, либо сначала зафиксировать planned migration в docs.
