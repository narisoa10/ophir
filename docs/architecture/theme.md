# Theme

## Назначение

Theme определяет единый способ доступа к визуальным значениям Ophir:
цветам, типографике, spacing, radii, shadows и другим design tokens. Feature-код
не должен знать конкретные числовые значения.

Этот документ является владельцем общей token architecture. Детальные правила
для текста, отступов и цветов находятся в отдельных документах:

- `typography_standard.md` - роли текста;
- `spacing_standard.md` - роли расстояний;
- `color_standard.md` - роли цветов.

Если возникает противоречие, детальный документ является источником истины для
своей области, а `theme.md` остается источником истины для общего правила:
feature-код получает значения только через Theme.

## Что такое Design Tokens

Design Token - именованное значение design system, используемое вместо raw
значения. Token описывает роль значения, а не его случайное число.

Пример: `spacing.md` лучше, чем `EdgeInsets.all(16)`, потому что роль `md`
можно централизованно изменить без ручного поиска по feature-коду.

## Группы токенов

- Color tokens: primary, secondary, surface, background, semantic colors.
- Typography tokens: роли текста, описанные в `typography_standard.md`.
- Spacing tokens: `xs`, `sm`, `md`, `lg`, `xl`.
- Radius tokens: стандартные скругления для containers, cards, inputs, buttons.
- Elevation/shadow tokens: тени только для элементов, где глубина несет смысл.
- Border tokens: stroke width и border color через Theme.

## Обязательное правило

Feature-код использует только Theme и роли. Конкретные значения находятся в
централизованном theme layer.

Raw значения допустимы только внутри реализации самого Theme layer. Там они
являются источником token values, а не feature-level styling.

## Запрещено в feature-коде

- `Color(...)`
- `EdgeInsets(...)`
- `BorderRadius(...)`
- `fontSize(...)`
- `FontWeight(...)`
- raw hex colors
- локальные typography helpers
- feature-specific theme classes

Исключение возможно только для прототипа, который не попадает в production
ветку. Перед merge raw значения должны быть заменены на tokens.

## Как использовать

1. Найти существующий token по роли.
2. Если роль подходит, использовать его без создания нового значения.
3. Если роль не подходит, проверить соседние документы: typography, spacing,
   color.
4. Если нового token действительно не хватает, обновить документацию и Theme.

## Когда создавать новый token

Новый token создается, если:

- значение повторится в нескольких местах;
- оно выражает самостоятельную роль в design system;
- без него feature-код будет использовать raw значение;
- изменение этого значения в будущем должно быть централизованным.

## Когда использовать существующий token

Существующий token используется, если отличие только визуальное, но роль та же.
Например, два section headers на разных экранах должны использовать одну роль,
если они выполняют одну функцию в иерархии.

## Как расширять

Расширение Theme требует:

1. Описать новую роль в соответствующем architecture document.
2. Обновить ADR, если решение меняет стандарт.
3. Добавить token в Theme.
4. Заменить raw или локальные значения в feature-коде.
5. Проверить визуальную консистентность на существующих экранах.

## Чего делать нельзя

- Создавать token под конкретный экран, например `operationsPageBlue`.
- Называть token по числу, например `spacing16`.
- Добавлять color без semantic или functional роли.
- Дублировать typography только ради небольшого отличия размера.

## Текущее ограничение

Theme является целевым стандартом. Если в существующей реализации есть
legacy visual values или параллельные theme modules, они должны рассматриваться
как миграционное состояние. Новые feature не должны расширять legacy-подход:
они обязаны использовать целевую token architecture.
