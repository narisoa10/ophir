# Category Architecture

## Назначение

Категории в Ophir должны одновременно поддерживать понятный UI и корректную
аналитику. Для этого `Category` имеет две независимые классификации:
`uiGroup` и `analyticsGroup`.

Этот документ является источником истины для category model, classification
rules и правил добавления новых категорий.

Место Category в общей доменной модели описано в `domain_model.md`. Этот документ
владеет только classification rules и правилами расширения категорий.

## Модель

```text
Category
├── id
├── name
├── icon
├── uiGroup
└── analyticsGroup
```

## Правило разделения

`uiGroup` отвечает только за группировку в интерфейсе.

`analyticsGroup` отвечает только за финансовую аналитику.

UI никогда не строит аналитику на основании `uiGroup`.

Analytics никогда не строит продуктовые выводы на основании `uiGroup`.

## Схема

```text
                 ┌──────────────┐
                 │   Category   │
                 └──────┬───────┘
                        │
          ┌─────────────┴─────────────┐
          │                           │
     ┌────▼─────┐              ┌──────▼────────┐
     │ uiGroup  │              │ analyticsGroup│
     └────┬─────┘              └──────┬────────┘
          │                           │
          ▼                           ▼
   Category picker,             Statistics,
   filters, labels,             budget, forecast,
   visual grouping              financial health
```

## Почему принято это решение

Одна классификация не может одинаково хорошо обслуживать интерфейс и аналитику.
Например, категория может быть удобна пользователю в группе "Daily", но для
аналитики относиться к "Needs" или "Variable expenses". Если смешать эти роли,
любое улучшение UI будет ломать статистику, а любое аналитическое изменение
будет менять поведение экранов.

## Как добавить новую категорию

1. Определить пользовательское название и icon.
2. Назначить `uiGroup` по тому, где категория должна отображаться в UI.
3. Назначить `analyticsGroup` по финансовому смыслу.
4. Проверить, что новая категория корректно отображается в picker.
5. Проверить, что аналитика получает ожидаемую группу.
6. Обновить документацию, если добавляется новая группа.

## Как расширять

Новые `uiGroup` добавляются только при изменении пользовательской навигации или
визуальной группировки.

Новые `analyticsGroup` добавляются только при появлении нового аналитического
смысла, который влияет на statistics, budget, forecast или financial health.

## Ошибки, которые нельзя допускать

- Использовать `uiGroup` для расчетов.
- Использовать `analyticsGroup` для порядка или внешнего вида picker.
- Создавать группу под одну категорию без долгосрочной причины.
- Дублировать категорию ради разных аналитических смыслов.
- Переименовывать группы без проверки влияния на данные и Supabase.

## Текущее ограничение

Если существующие данные или mapping еще содержат старую группу категорий, это
должно рассматриваться как migration issue. Новые категории должны создаваться
только с явным разделением `uiGroup` и `analyticsGroup`.
