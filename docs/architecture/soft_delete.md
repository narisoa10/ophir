# Soft Delete

## Назначение

Ophir использует soft delete для финансовых данных. Удаление операции не должно
немедленно уничтожать историю, потому что финансовые записи влияют на баланс,
аналитику, восстановление ошибок и доверие пользователя.

Этот документ является источником истины для lifecycle архивирования и будущего
permanent delete.

## Жизненный цикл

```text
Created
↓
Active
↓
Archived
↓
Deleted permanently (future)
```

## Состояния

### Created

Запись создана локально или получена от пользователя, но еще может проходить
валидацию или сохранение.

### Active

Запись участвует в обычных списках, running balance и аналитике.

### Archived

Запись скрыта из основных экранов, но остается в базе. Для этого используется
`archived_at`.

### Deleted permanently

Будущее состояние для окончательного удаления. Оно должно быть доступно только
после отдельного product decision и реализации Archive Screen.

## Почему используется archived_at

`archived_at` фиксирует момент архивирования и позволяет:

- отличать активные записи от архивных;
- показывать историю архивирования;
- восстанавливать запись;
- строить будущий Archive Screen;
- избежать потери данных при случайном удалении.

## Почему отказались от hard delete

Hard delete опасен для финансового приложения:

- пользователь может случайно удалить важную операцию;
- аналитика может резко измениться без объяснения;
- синхронизация сложнее восстанавливается после ошибки;
- невозможно построить надежный UX восстановления.

## Как будет работать восстановление

Восстановление должно очищать `archived_at` и возвращать запись в `Active`.
После восстановления operation снова участвует в списках, running balance и
аналитике согласно domain rules.

## План Archive Screen

Будущий Archive Screen должен:

- показывать archived operations;
- поддерживать filter по дате, категории и типу;
- давать restore action;
- отделять restore от permanent delete;
- предупреждать пользователя перед final delete;
- не смешивать archived items с основным Operations screen.

## Как использовать

Любой query для основных экранов должен исключать archived records, если feature
явно не работает с архивом.

## Как расширять

Permanent delete добавляется только после ADR, потому что это меняет гарантию
сохранности финансовых данных.

## Запрещено

- Удалять финансовые записи hard delete из обычного UI.
- Считать archived records активными в running balance.
- Скрывать archived behavior внутри repository без domain-level правила.
- Реализовывать restore как создание новой операции.

## Текущее ограничение

Если Archive Screen или permanent delete еще не реализованы, это не отменяет
обязательность `archived_at`. Новые удаления должны проектироваться как archive
action, а не как hard delete.
