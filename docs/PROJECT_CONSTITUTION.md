# PROJECT CONSTITUTION

Version: 1.1

Этот документ обязателен для всех изменений в проекте **Ophir**.

Каждый Stage должен начинаться с выполнения правил данного документа.

Если Stage явно переопределяет правило, приоритет имеет Stage только в этой части.

---

# 1. Preflight

Перед началом любого Stage обязательно:

1. Проверить текущую директорию (`pwd` / `Get-Location`).

2. Выполнить:

```bash
git status --short
```

Если Git недоступен или открыта неправильная директория — остановиться и сообщить.

---

# 2. Working Tree Policy

Перед началом Stage определить состояние рабочего дерева.

### Clean

Если дерево чистое — продолжить работу.

### Dirty

Если обнаружены изменения:

- перечислить все файлы;
- определить, относятся ли они к текущему Stage.

Если изменения НЕ относятся к текущему Stage:

- остановить выполнение;
- не выполнять `git add .`;
- запросить решение пользователя.

Допустимые варианты:

- отдельный Git Checkpoint;
- исключить изменения;
- явно разрешить объединение.

Никогда автоматически не смешивать изменения разных Stage.

---

# 3. Scope

Изменять только Scope текущего Stage.

Запрещено:

- лишний рефакторинг;
- изменение соседних модулей;
- изменение архитектуры без необходимости;
- побочные улучшения.

---

# 4. Архитектура

Сохранять существующую архитектуру проекта.

Основные принципы:

- Domain содержит бизнес-логику.
- UI только отображает данные.
- Dashboard — consumer.
- Не создавать provider cycles.
- Не нарушать направление зависимостей.
- Не создавать скрытых зависимостей между слоями.

---

# 5. Runtime Safety

Любое изменение runtime должно:

- быть минимальным;
- быть обратимым;
- иметь rollback;
- сопровождаться тестами.

Не изменять пользовательское поведение без явного требования Stage.

---

# 6. Legacy Policy

Legacy не удаляется во время обычной реализации.

Удаление допускается только отдельным Stage после:

- audit;
- regression tests;
- parity;
- явного разрешения.

---

# 7. Coding Standards

Следовать существующему стилю проекта.

Обязательно:

- один файл = один public class;
- один файл = один public enum;
- один файл = один public extension;
- один файл = один public typedef.

Не оставлять:

- TODO;
- заглушки;
- пустые реализации;
- закомментированный код.

Изменения должны быть минимально необходимыми.

---

# 8. Theme V1

Весь UI использует Theme V1.

Использовать Theme V1 для:

- colors;
- typography;
- spacing;
- radius;
- dimensions;
- elevation.

Запрещено использовать:

- magic numbers;
- прямые Colors.*;
- прямые TextStyle;
- прямые EdgeInsets;
- прямые BorderRadius;

если существует Theme V1 эквивалент.

---

# 9. Localization

Все пользовательские строки должны использовать l10n.

Поддерживаемые языки:

- English
- Français
- Русский

Строковые литералы в UI запрещены.

---

# 10. Protected Areas

Без отдельного Scope запрещено изменять:

- Supabase;
- persistence;
- Dashboard UI;
- Operations contracts;
- Categories contracts;
- financial calculations;
- публичные API.

---

# 11. Tests

После любых изменений обязательно выполнить:

```bash
flutter analyze
```

```bash
flutter test test/features/assistant
```

```bash
flutter test test/features/dashboard
```

```bash
flutter test test/features/categories
```

Если проверки не прошли:

Stage считается незавершённым.

---

# 12. Final Report

После каждого Stage предоставить:

- Preflight;
- список изменённых файлов;
- описание изменений;
- Runtime / Architecture Audit (если применимо);
- Remaining Legacy (если применимо);
- Risks / TODO;
- результаты analyze;
- результаты тестов;
- git status --short;
- Final Audit.

---

# 13. Git Checkpoint

Git Checkpoint выполняется только после успешного Stage.

Перед checkpoint обязательно убедиться, что рабочее дерево содержит только изменения текущего Stage.

Выполнить:

```bash
git add .
git commit -m "<meaningful message>"
git push
```

После этого предоставить:

- commit hash;
- commit message;
- git push result;
- git status;
- git log --oneline -5;

Подтвердить:

- Working tree clean.
- origin/main updated.

---

# 14. Quality Gate

Stage считается завершённым только если:

- Scope выполнен полностью;
- архитектурные границы соблюдены;
- provider cycles отсутствуют;
- runtime regressions отсутствуют;
- analyze зелёный;
- тесты зелёные;
- Git Checkpoint выполнен.

---

# 15. Communication

Если обнаружен blocker:

- остановиться;
- не придумывать обходное решение самостоятельно;
- описать проблему;
- предложить безопасные варианты действий.

Не продолжать Stage, если существует риск нарушения архитектуры, потери данных или смешивания изменений.

---

# Использование

Каждый Stage начинается одной строкой:

> Перед выполнением Stage обязательно прочитай и строго соблюдай `docs/PROJECT_CONSTITUTION.md`.

После этого выполняется только Scope текущего Stage.
